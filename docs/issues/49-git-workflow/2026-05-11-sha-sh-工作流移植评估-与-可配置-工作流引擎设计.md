# sha.sh 开发工作流移植评估与可配置工作流引擎设计

## 1. 背景

`sha.sh` 是 xx 项目目前使用的开发工作流脚本，基于 bash + gh CLI + jq 实现了一套完整的 **worktree 隔离开发模式** 工作流。该脚本共有约 750 行 bash，包含以下几大模块：

| 模块 | 子命令 | 行数 | 复杂度 |
|------|--------|------|--------|
| `dev` | list, pr, status, merge_pr, remove | ~260 | 高 |
| `issue` | new, dev, list | ~100 | 中 |
| `ws` | exec, ci, test, check, fix | ~30 | 低 |
| `submodule` | status, exec | ~20 | 低 |
| `sync` | nodejs, submodule, xx, all | ~20 | 低 |
| `clean` | — | ~5 | 低 |

### 1.1 移植动机

1. **bash 力不从心**：复杂 JSON 处理需多趟遍历、临时文件、易碎的分隔符解析
2. **无法被复用**：脚本深度耦合了 xx 项目的特定配置（GH_OWNER, GH_PROJECT_NUM 等）
3. **无类型安全**：变量名拼错就运行时崩溃
4. **不可测试**：依赖 gh/git 外部进程，无法 mock
5. **不可组合**：bash 函数无法被其他程序 import 使用

### 1.2 目标

在 `packages/xx-cli/src/common/workflow.ts`（当前为空文件）实现**可配置、可复用、可测试**的工作流引擎，形成标准接口，然后通过配置注入项目特定信息。其他项目只需一段配置即可复用全部工作流逻辑。

## 2. 现有流程拆解

### 2.1 核心流程

```
issue new → issue dev → (开发) → dev pr → (合并) → dev remove
```

每一步在实际执行的底层操作：

| 步骤 | 底层操作 |
|------|----------|
| `issue new "描述"` | `gh issue create` → `gh project item-add` |
| `issue dev 42 [名称]` | `gh project item-edit Status→In progress` → `git worktree add` → `git push` |
| (开发) | 编码、commit |
| `dev pr` | 提取 issue 编号 → `gh issue view` → `git push` → `gh pr create` |
| `dev merge_pr` | `gh pr list` → `gh pr merge --squash` |
| `dev remove` | `git checkout main` → `git pull` → `git worktree remove` → `git branch -d` |

### 2.2 辅助命令

| 命令 | 功能 |
|------|------|
| `dev list` | 所有 worktree 总览（分支/PR/Project/ahead） |
| `dev status` | 当前分支详情（Git/Issue/PR/Project 四源合并） |
| `issue list` | 列出未完成任务（过滤 Backlog/Done） |

### 2.3 分支名约定

支持三种格式从分支名提取 issue 编号：

```
42-feat-name    → 42
issue-42        → 42
42              → 42
```

## 3. 适配性分析：另一个项目能否直接用？

### 3.1 核心流程本身是通用的

```
issue new → issue dev → (开发) → dev pr → (合并) → dev remove
```

这套流程是基于 **GitHub Issues + GitHub Projects + Worktree 隔离开发** 的标准组合，任何一个使用 GitHub Projects 做看板、GitHub Issues 管理任务、worktree 做分支隔离的 TS 项目都可以套用。

### 3.2 需要抽象/改造的 7 个关键点

#### 3.2.1 🔴 分支名提取 Issue 编号的逻辑是隐式约定

当前 xx 支持 `42-feat-name` / `issue-42` / `42` 三种格式。但其他项目可能有完全不同规则：

- `feat/42-something`（GitLab/Conventional 风格）
- `feature/PROJ-42-feat-name`（JIRA 风格）
- `fix/` + issue 编号后续一致
- 甚至**根本不用 issue 编号作分支名**（直接分支名匹配 PR）

**改造要求**：分支名解析必须策略化、可注入。

#### 3.2.2 🔴 Project 自定义字段名是硬编码的

当前硬编码了字段名 `"Status"`、选项名 `"In progress"`、以及读取 `priority`/`module` 字段。但不同项目：

- 字段名可能是 `Stage` 而非 `Status`
- 选项名可能是 `Doing` 而非 `In progress`
- 可能没有 `Priority`、`Module` 字段
- 选项名可能含中文

**改造要求**：字段映射必须可配置。

#### 3.2.3 🔴 worktree 目录名不可改

`_worktree_dir=".worktree"` 写死在脚本里。其他项目可能希望用 `.worktrees/`、`work/` 等。

**改造要求**：目录名配置化。

#### 3.2.4 🔴 合并策略只有 squash

```bash
gh pr merge "$pr_num" --repo "$GH_REPO" --squash
```

团队偏好不一是常见情况（squash / merge commit / rebase）。

**改造要求**：合并策略可配置。

#### 3.2.5 🔴 gh CLI 是硬依赖，无抽象层

全部通过调用 `gh` CLI 实现。这意味着：

- **无法适配 GitLab**（必须用 `glab`）
- **无法单元测试**（无法 mock）

**改造要求**：封装 `GhProvider` 接口，所有外部调用走接口，支持 mock。

#### 3.2.6 🟡 缺少 `Backlog → Ready` 过渡环节

当前流程：

```
issue new → issue dev
  Backlog    In progress
```

但 Scrum/看板的完整状态流是：

```
Backlog → Ready → In progress → In review → Done
```

`issue new` 创建后直接进入 Backlog。`issue list` 只显示 Ready/In progress/In review。**但没有命令将 Backlog 中的 issue 推进到 Ready**（通常由 backlog refinement 完成）。

**改造要求**：补充 `issue ready <num>` 或保持灵活——有些项目通过 GitHub Actions 自动做，有些没有这个环节。

#### 3.2.7 🟡 `dev list` 假设每个 branch 都对应一个 issue

当 worktree 分支找不到对应 issue 时会静默忽略字段（issue_status 留空）。但不会报错——这个处理方式本身是好的。问题在于**未明确文档化**，且用户无法配置"无 issue 的 branch 是否应该标红警告"。

### 3.3 可保留原样的部分

| 逻辑 | 原因 |
|------|------|
| `_get_main_branch()` 自动检测 main/master | 通用，保持 |
| `_get_gh_user()` | gh 跨项目行为一致 |
| `dev list` 的颜色/列宽格式 | 工具内嵌默认格式，用户可后续定制 |
| `ws` / `submodule` 批量执行 | 简单 bash 包装，移植收益小，暂不处理 |
| `sync all` | 已由 xx-cli 的 `syncDependencies()` 独立实现 |

## 4. 配置接口设计

### 4.1 完整配置定义

```typescript
export interface WorkflowConfig {
  /** GitHub 项目身份 */
  gh: {
    owner: string;
    repo: string;
    projectNum: number;
    projectId: string;
  };

  /** Git 工作目录配置 */
  git: {
    /** worktree 子目录名，默认 ".worktree" */
    worktreeDir?: string;
    /** 主分支名，不填则自动检测 main/master */
    mainBranch?: string;
  };

  /** 分支名 → issue 编号解析策略 */
  branchParser?: {
    parseIssueNumber(branch: string): string | undefined;
  };

  /** Project 自定义字段映射 */
  projectFields?: {
    status: {
      fieldName: string;          // 默认 "Status"
      options: {
        backlog: string;          // 默认 "Backlog"
        ready: string;            // 默认 "Ready"
        inProgress: string;       // 默认 "In progress"
        inReview: string;         // 默认 "In review"
        done: string;             // 默认 "Done"
      };
    };
    /** Priority 字段名，设为空字符串则不读取 */
    priorityField?: string;       // 默认 "Priority"
    /** Module 字段名，设为空字符串则不读取 */
    moduleField?: string;         // 默认 "Module"
  };

  /** issue list 时关注的状态列表 */
  issueStatusFilter?: string[];   // 默认 ["Ready", "In progress", "In review"]

  /** PR 合并策略 */
  mergeMethod?: "squash" | "merge" | "rebase";  // 默认 "squash"

  /** 无 issue 关联的 branch 处理行为 */
  onBranchWithoutIssue?: "warn" | "silent";  // 默认 "silent"
}
```

### 4.2 默认配置提供器

```typescript
export function defaultWorkflowConfig(partial: WorkflowConfig): Required<WorkflowConfig> {
  return {
    gh: partial.gh,
    git: {
      worktreeDir: partial.git?.worktreeDir ?? ".worktree",
      mainBranch: partial.git?.mainBranch,
    },
    branchParser: partial.branchParser ?? {
      parseIssueNumber(branch: string): string | undefined {
        // 兼容 42-feat-name / issue-42 / 42 三种格式
        const patterns = [
          /^(\d+)-/,           // 42-feat-name
          /^issue-(\d+)$/,     // issue-42
          /^(\d+)$/,           // 42
        ];
        for (const re of patterns) {
          const m = branch.match(re);
          if (m) return m[1];
        }
        return undefined;
      },
    },
    projectFields: partial.projectFields ?? {
      status: {
        fieldName: "Status",
        options: {
          backlog: "Backlog",
          ready: "Ready",
          inProgress: "In progress",
          inReview: "In review",
          done: "Done",
        },
      },
      priorityField: "Priority",
      moduleField: "Module",
    },
    issueStatusFilter: partial.issueStatusFilter ?? ["Ready", "In progress", "In review"],
    mergeMethod: partial.mergeMethod ?? "squash",
    onBranchWithoutIssue: partial.onBranchWithoutIssue ?? "silent",
  };
}
```

### 4.3 使用示例

**xx 项目自身**（只需注入配置，0 代码改动工作流逻辑）：

```typescript
import { createWorkflow } from "@xx-agent/xx-cli/workflow.js";

const wf = createWorkflow({
  gh: {
    owner: "xx-agent",
    repo: "xx-agent/xx",
    projectNum: 2,
    projectId: "PVT_kwDOENC_Tc4BXDLA",
  },
  git: {
    worktreeDir: ".worktree",
  },
  mergeMethod: "squash",
});

// issue 工作流
await wf.issue.new("支持插件热加载");        // create + add to project
await wf.issue.dev(42, "hot-reload");        // Status→In progress + worktree
await wf.issue.list();                       // 列出未完成任务

// dev 工作流
await wf.dev.list();                         // 所有 worktree 总览
await wf.dev.status();                       // 当前 worktree 详情
await wf.dev.pr();                           // push + create PR
await wf.dev.mergePr();                      // squash merge
await wf.dev.remove();                       // cleanup worktree + branch
```

**其他项目使用**（改配置适配）：

```typescript
import { createWorkflow } from "@xx-agent/xx-cli/workflow.js";

const wf = createWorkflow({
  gh: {
    owner: "my-org",
    repo: "my-org/my-project",
    projectNum: 5,
    projectId: "PVT_kwHOMyProject1234",
  },
  git: {
    worktreeDir: "my-worktrees",
  },
  // 自定义分支规则：feat/PROJ-42-foo
  branchParser: {
    parseIssueNumber: (branch) => {
      const m = branch.match(/^[a-z]+\/(PROJ-\d+)/i);
      return m?.[1];
    },
  },
  // 自定义字段
  projectFields: {
    status: {
      fieldName: "Stage",
      options: {
        backlog: "Backlog",
        ready: "Selected",
        inProgress: "In Progress",
        inReview: "Review",
        done: "Closed",
      },
    },
    priorityField: "",    // 没有 Priority 字段
    moduleField: "Area",  // 叫 Area 不叫 Module
  },
  mergeMethod: "rebase",
});
```

## 5. 文件结构

```
packages/xx-cli/src/common/workflow/
├── index.ts        # createWorkflow() + re-export
├── types.ts        # WorkflowConfig, WorktreeInfo, ProjectItem, IssueInfo, PRInfo
├── config.ts       # defaultWorkflowConfig() + 配置合并
├── github.ts       # GhProvider 接口 + GhCliProvider 实现
├── git.ts          # GitProvider 接口 + GitCliProvider 实现
├── dev.ts          # DevWorkflow 类（list, pr, status, mergePr, remove）
└── issue.ts        # IssueWorkflow 类（new, dev, list）
```

### 文件职责

| 文件 | 导出 | 职责 |
|------|------|------|
| `types.ts` | `WorkflowConfig`, `WorktreeInfo`, `ProjectItem`, `IssueInfo`, `PRInfo`, `Provider` 接口 | 共享类型和外部依赖抽象 |
| `config.ts` | `defaultWorkflowConfig()` | 默认值补全 + 配置合并 |
| `github.ts` | `GhProvider` 接口 + `GhCliProvider` | 封装 `gh` CLI 调用（issue/project/pr） |
| `git.ts` | `GitProvider` 接口 + `GitCliProvider` | 封装 `git` CLI 调用（worktree/branch/status） |
| `dev.ts` | `DevWorkflow` | dev 工作流编排 |
| `issue.ts` | `IssueWorkflow` | issue 工作流编排 |
| `index.ts` | `createWorkflow()` | 工厂函数，构建所有工作流对象并暴露 |

## 6. 与 xx-cli CLI 的集成

对应的 commander 子命令可以这样挂载：

```typescript
// packages/xx-cli/src/workflow-cli.ts

import { createWorkflow } from "./common/workflow/index.js";

// 从项目 xx.yaml 或环境变量读取配置
const config = loadWorkflowConfig();

// 也可以允许命令行参数覆盖
const wf = createWorkflow(config);

const workflowCli = new Command("workflow")
  .description("GitHub Project 工作流管理");

// issue 子命令
const issueCmd = new Command("issue");
issueCmd
  .command("new <description...>")
  .action(async (desc: string[]) => { await wf.issue.new(desc.join(" ")); });
issueCmd
  .command("dev <issueNum> [branchName]")
  .action(async (num: string, name?: string) => { await wf.issue.dev(parseInt(num), name); });
issueCmd
  .command("list")
  .action(async () => { await wf.issue.list(); });
workflowCli.addCommand(issueCmd);

// dev 子命令
const devCmd = new Command("dev");
devCmd
  .command("list")
  .action(async () => { await wf.dev.list(); });
devCmd
  .command("status")
  .action(async () => { await wf.dev.status(); });
devCmd
  .command("pr")
  .action(async () => { await wf.dev.pr(); });
devCmd
  .command("merge-pr")
  .action(async () => { await wf.dev.mergePr(); });
devCmd
  .command("remove")
  .action(async () => { await wf.dev.remove(); });
workflowCli.addCommand(devCmd);

export { workflowCli };
```

然后在 `cli.ts` 挂载：

```typescript
program.addCommand(workflowCli);
```

## 7. 阶段性建议

### 第一阶段（核心，可提交）

实现 `types.ts` + `config.ts` + `github.ts` + `git.ts` + `dev.ts` + `issue.ts` + `index.ts`，覆盖 `dev` 和 `issue` 工作流的全部子命令。在 `cli.ts` 挂载 `workflow` 子命令树。

交付后可替换掉 `sha.sh` 中 70% 的工作流逻辑（~400 行 bash）。

### 第二阶段（增强，可选）

- 补充单元测试（mock GhProvider/GitProvider）
- 支持 `xx.yaml` 读取 WorkflowConfig
- 添加 `workflow init` 命令生成默认配置
- 支持 `--dry-run` 模式预览操作但不执行

### 第三阶段（扩展，可选）

- GitLab 适配（`GlabProvider` 实现同一接口）
- 支持自定义表格输出格式（JSON / markdown / plain text）
- CI 集成模式（非交互式工作流）

## 8. 风险评估

| 风险 | 缓解措施 |
|------|----------|
| `gh` CLI 版本差异导致 JSON 输出格式不同 | 封装层捕获格式错误，提供友好错误信息 |
| `git worktree list` 输出格式在 Mac/Linux 上有差异 | 使用 `--porcelain` 格式保证跨平台一致 |
| 配置过于灵活导致维护负担 | 提供完整的默认值，90% 项目只需要填 3 个必填字段 |
| 迁移期间 sha.sh 和 TS 工作流并行存在 | 两者的命令入口明确区分（`./sha.sh` vs `npx xx-cli workflow`） |
